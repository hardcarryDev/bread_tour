## F1. Stamp Tour (GPS 자동 스탬프 + 취소/정정)

### AC-F1-01 — dwell-time 충족 시 자동 스탬프 + 시각 기록 (REQ-F1-002, REQ-F1-003)
- **Given** 멤버가 로그인하여 투어를 진행 중이고, 장소 A에 좌표와 반경 50m가 등록되어 있고, 위치 권한과 정확도 조건이 충족된 상태에서
- **When** 멤버의 위치가 장소 A 반경(50m) 안에서 dwell-time 조건(연속 3샘플 또는 누적 10초, A9)을 충족하면
- **Then** 장소 A의 스탬프가 자동 적용되고, 서버 기준 도착 시각이 기록되어 화면에 표시된다.

### AC-F1-02 — 중복 스탬프 방지 (REQ-F1-004)
- **Given** 멤버가 이미 장소 A의 유효한 스탬프를 보유한 상태에서
- **When** 멤버가 장소 A의 반경에 다시 진입하여 도착 조건을 충족해도
- **Then** 새 스탬프가 추가되지 않고, 최초 도착 시각이 그대로 유지된다.

### AC-F1-03 — 정확도 부족 시 자동 스탬프 보류 (REQ-F1-006)
- **Given** 위치 권한은 허용되었으나 보고된 정확도(accuracy)가 반경×0.5(50m 기준 25m) 이상인 상태에서
- **When** 위치가 장소 반경 근처로 들어와도
- **Then** 자동 스탬프는 적용되지 않고, 사용자에게 위치 정확도 경고가 표시된다.

### AC-F1-04 — 수동 체크인 폴백 (REQ-F1-007)
- **Given** GPS 자동 스탬프가 불가능한 상태(권한 거부 또는 정확도 부족)에서
- **When** 멤버가 수동 체크인을 요청하고 다른 멤버의 확인이 이루어지면
- **Then** 해당 장소의 스탬프가 수동(method=manual)으로 기록된다.

### AC-F1-05 — 반경 겹침 시 방문 순서 우선 (REQ-F1-008)
- **Given** 장소 A(순서 1, 미스탬프)와 장소 B(순서 2, 미스탬프)의 반경이 겹치는 지점에 멤버가 있는 상태에서
- **When** 그 한 지점에서 도착 조건이 충족되면
- **Then** 방문 순서가 더 빠른 장소 A에만 스탬프가 적용되고, A와 B에 동시에 자동 스탬프가 부여되지 않는다. (순서 동순위/미정 시 중심거리가 더 가까운 장소를 택한다.)

### AC-F1-06 — 장소 좌표/반경 저장 (REQ-F1-001)
- **Given** 멤버가 장소를 등록하는 상태에서
- **When** 카카오맵 검색 또는 핀 지정으로 좌표를 선택하고 저장하면
- **Then** 해당 장소에 위도/경도와 도착 판정 반경(기본 50m)이 저장되고, 이후 조회 시 동일 값이 반환된다.

### AC-F1-07 — 진행 화면의 스탬프/시각 순서 표시 (REQ-F1-005)
- **Given** 투어에 장소 3곳이 순서대로 등록되고 일부에 스탬프가 있는 상태에서
- **When** 투어 진행 화면을 열면
- **Then** 각 장소의 스탬프 획득 여부와 도착 시각이 방문 순서대로 표시된다.

### AC-F1-08 — 스탬프 취소/정정 (REQ-F1-009)
- **Given** 멤버 본인 또는 투어 소유자가 잘못 찍힌 스탬프를 보고 있는 상태에서
- **When** 스탬프 취소(또는 도착 시각 정정)를 요청하면
- **Then** 해당 스탬프가 취소(또는 시각 정정) 처리되고 변경 사실이 기록되며, 진행 화면에 반영된다.

### AC-F1-09 — 권한 없는 취소 거부 (REQ-F1-010)
- **Given** 스탬프 소유자도 투어 소유자도 아닌 멤버가
- **When** 다른 멤버의 스탬프 취소/정정을 시도하면
- **Then** 요청이 거부되고 스탬프는 변경되지 않는다.

### AC-F1-10 — 취소 후 재스탬프 허용 (REQ-F1-011, D14)
- **Given** 멤버의 장소 A 스탬프가 취소된 상태에서
- **When** 동일 멤버가 장소 A 반경에서 다시 도착 조건을 충족하면
- **Then** 새 스탬프가 정상 적용된다(UNIQUE 제약이 재스탬프를 막지 않음, plan.md 참조).

## F2. Directions / 길찾기

### AC-F2-01 — 두 장소 간 경로 + 거리/시간 표시 (REQ-F2-001, REQ-F2-002)
- **Given** 투어에 장소 A, B가 등록된 상태에서
- **When** 멤버가 A→B 길찾기를 요청하면
- **Then** 카카오맵에 경로가 표시되고, 경로 거리(미터)와 예상 소요 시간(분, 카카오 SDK 제공 이동 수단 기준)이 함께 표시된다.

### AC-F2-02 — 다음 장소 안내 (REQ-F2-003)
- **Given** 멤버가 위치 권한을 허용하고 미방문 장소가 남아 있는 상태에서
- **When** "다음 장소로 안내"를 선택하면
- **Then** 현재 위치에서 방문 순서상 다음 미방문 장소까지의 경로가 안내된다.

### AC-F2-03 — 경로 API 실패 폴백 (REQ-F2-004)
- **Given** 카카오맵 경로 API가 오류를 반환하는 상태에서
- **When** 멤버가 길찾기를 요청하면
- **Then** 오류 메시지가 표시되고, 지도와 장소 마커는 계속 정상적으로 보인다.

## F3. Map Ordering (지도 순서 표시)

### AC-F3-01 — 순번 마커 표시 (REQ-F3-001, REQ-F3-002)
- **Given** 투어에 빵집/음식점 3곳이 순서대로 등록된 상태에서
- **When** 지도 화면을 열면
- **Then** 각 장소가 마커로 표시되고 방문 순서 번호(1,2,3)가 마커에 함께 표시된다.

### AC-F3-02 — 순서 변경 시 재렌더 (REQ-F3-003)
- **Given** 지도에 순번이 표시된 상태에서
- **When** 멤버가 방문 순서를 변경하면
- **Then** 지도의 번호와 연결 경로가 변경된 순서로 즉시 다시 그려진다.

### AC-F3-03 — 마커 선택 요약 (REQ-F3-004)
- **Given** 지도에 장소 마커가 표시된 상태에서
- **When** 멤버가 특정 마커를 선택하면
- **Then** 해당 장소의 이름, 순서, 추천 메뉴, 스탬프 상태가 요약 표시된다.

## F4. Menu Recommendation (메뉴 추천)

### AC-F4-01 — 추천 메뉴 입력/표시 (REQ-F4-001, REQ-F4-002)
- **Given** 멤버가 빵집을 등록/편집하는 상태에서
- **When** 시그니처/추천 메뉴를 입력하고 저장하면
- **Then** 해당 빵집 상세에 입력한 추천 메뉴가 표시된다.

### AC-F4-02 — 멤버별 추천 병행 표시 (REQ-F4-003)
- **Given** 한 빵집에 멤버 X, Y가 각각 추천 메뉴를 입력한 상태에서
- **When** 해당 빵집 상세를 열면
- **Then** 두 멤버의 추천 메뉴가 입력자 정보와 함께 모두 표시된다.

### AC-F4-03 — 추천 메뉴 미입력 허용 (REQ-F4-004)
- **Given** 멤버가 빵집을 등록하면서 추천 메뉴를 비워둔 상태에서
- **When** 빵집을 저장하면
- **Then** 빵집이 정상 등록되고 상세에 "추천 메뉴 없음"으로 표시된다.

## F5. Collaborative Editing (실시간 공동 편집 + 인증)

### AC-F5-01 — 인증 멤버만 접근 (REQ-F5-001, REQ-F5-006)
- **Given** 투어가 특정 멤버들에게만 공유된 상태에서
- **When** 초대되지 않은/미인증 사용자가 투어 접근을 시도하면
- **Then** 접근이 거부되고 인증/초대 안내가 표시된다.

### AC-F5-02 — 실시간 변경 반영 (REQ-F5-002, NFR-PERF-002)
- **Given** 멤버 X와 Y가 같은 투어를 동시에 보고 있는 상태에서
- **When** X가 장소를 추가/삭제하거나 순서를 변경하면
- **Then** Y의 화면에 통상 2초 이내로 동일한 변경이 반영된다.

### AC-F5-03 — 접속 멤버 표시 (REQ-F5-003)
- **Given** 여러 멤버가 같은 투어에 접속한 상태에서
- **When** 투어 화면을 보면
- **Then** 현재 접속 중인 멤버 목록(presence)이 표시된다.

### AC-F5-04 — 동시 편집 행 단위 충돌 해결 (REQ-F5-004, NFR-CONFLICT-002)
- **Given** 멤버 X, Y가 동일한 레코드(예: 같은 장소 행)를 거의 동시에 편집하는 상태에서
- **When** 두 변경이 서버에 도달하면
- **Then** 서버 타임스탬프가 더 늦은 변경이 그 행의 최종값으로 확정되고(행 단위 last-write-wins), 덮어쓰인 멤버에게 최신 값이 갱신되었다는 비파괴적 알림이 표시된다.

### AC-F5-05 — 오프라인/재연결 동기화 (REQ-F5-005)
- **Given** 멤버가 편집 화면을 보던 중 네트워크가 끊긴 상태에서
- **When** 다시 온라인으로 복귀하면
- **Then** 마지막 동기화 상태에서 서버 기준 최신 상태로 동기화된다.

### AC-F5-06 — 동시 순서 변경 일관성 (REQ-F5-007)
- **Given** 멤버 X, Y가 거의 동시에 방문 순서를 변경하는 상태에서
- **When** 두 변경이 서버에 도달하면
- **Then** 서버 측 단일 트랜잭션 재계산으로 order_index가 1..N의 중복/누락 없는 연속 값으로 확정된다.

## F6. Tour Lifecycle & Permissions (투어 생명주기 및 권한)

### AC-F6-01 — 투어 생성 시 소유자 지정 (REQ-F6-001)
- **Given** 인증된 사용자가
- **When** 새 투어를 생성하면
- **Then** 생성자가 해당 투어의 owner로 기록되고 소유자 권한을 가진다.

### AC-F6-02 — 멤버 초대 생성 (REQ-F6-002)
- **Given** 소유자(또는 권한 있는 멤버)가 투어를 보고 있는 상태에서
- **When** 멤버 초대를 요청하면
- **Then** 초대 링크 또는 이메일 초대 수단이 생성된다.

### AC-F6-03 — 초대 수락/거절 (REQ-F6-003)
- **Given** 초대받은 사용자가 초대 링크를 연 상태에서
- **When** 수락하면 멤버로 추가되고, 거절하면
- **Then** 멤버로 추가되지 않으며 투어 데이터에 접근할 수 없다.

### AC-F6-04 — 소유자 권한 (REQ-F6-004)
- **Given** 투어 소유자가 로그인한 상태에서
- **When** 장소 삭제 / 순서 변경 / 멤버 제거 / 투어 삭제를 시도하면
- **Then** 각 작업이 정상 수행된다.

### AC-F6-05 — 일반 멤버 권한 (REQ-F6-005)
- **Given** 일반 멤버가 로그인한 상태에서
- **When** 장소 추가/편집, 메뉴 입력, 자신의 스탬프 기록을 시도하면
- **Then** 각 작업이 정상 수행된다.

### AC-F6-06 — 소유자 전용 작업 거부 (REQ-F6-006)
- **Given** 일반 멤버가 로그인한 상태에서
- **When** 투어 삭제 또는 다른 멤버 제거를 시도하면
- **Then** 요청이 거부된다.

### AC-F6-07 — 장소 삭제 시 cascade + 순서 재계산 (REQ-F6-007)
- **Given** 장소 B(순서 2)에 추천 메뉴와 스탬프가 연결된 상태에서
- **When** 권한 있는 사용자가 장소 B를 삭제하면
- **Then** B에 연결된 spot_menus와 stamps가 함께 삭제되고, 남은 장소의 order_index가 빈틈 없이(1..N) 재계산된다.

## NFR Acceptance

### AC-NFR-GEO-01 — 권한 거부 폴백 (NFR-GEO-002)
- **Given** 사용자가 위치 권한을 거부한 상태에서
- **When** 앱을 사용하면
- **Then** 자동 스탬프는 비활성화되지만 지도/길찾기/계획/실시간 편집은 정상 동작하고, 수동 체크인 안내가 표시된다.

### AC-NFR-GEO-02 — 보안 컨텍스트 (NFR-GEO-003)
- **Given** 앱이 비 HTTPS(비보안 컨텍스트)로 로드된 상태에서
- **When** 위치 기능을 시도하면
- **Then** Geolocation 기능이 활성화되지 않는다.

### AC-NFR-GEO-03 — 권한 목적 안내 후 요청 (NFR-GEO-001)
- **Given** 위치 기능이 필요한 화면에 처음 진입한 상태에서
- **When** 화면이 표시되면
- **Then** 자동 스탬프 목적 안내가 먼저 표시된 뒤 브라우저 위치 권한 요청이 발생한다.

### AC-NFR-GEO-04 — 추적 활성 표시기 (NFR-GEO-004)
- **Given** 위치 추적(watchPosition)이 활성화된 상태에서
- **When** 투어 진행 화면을 보면
- **Then** 추적이 진행 중임을 나타내는 지속 표시기가 화면에 보인다.

### AC-NFR-GEO-05 — 추적 중지 컨트롤 (NFR-GEO-005)
- **Given** 위치 추적이 활성화된 상태에서
- **When** 사용자가 추적 일시중지/중지 컨트롤을 누르면
- **Then** 위치 추적이 중단되고, 중단 동안 자동 스탬프가 적용되지 않는다.

### AC-NFR-GEO-06 — 원시 좌표 미저장 (NFR-GEO-006)
- **Given** 멤버가 투어를 진행하여 여러 스탬프를 획득한 상태에서
- **When** stamps 및 관련 테이블을 점검하면
- **Then** 원시 위치 좌표 이력은 저장되어 있지 않고, 도착 시각과 장소 식별자만 기록되어 있다.

### AC-NFR-CONFLICT-01 — 변경 메타데이터 기록 (NFR-CONFLICT-001)
- **Given** 멤버가 일정 레코드를 변경한 상태에서
- **When** 해당 레코드를 조회하면
- **Then** 서버 타임스탬프(updated_at)와 변경 멤버 식별자가 함께 기록되어 있다.

### AC-NFR-CONFLICT-02 — 덮어쓰기 비파괴 알림 (NFR-CONFLICT-003)
- **Given** 멤버 X가 아직 반영되지 않은 로컬 변경을 가진 상태에서
- **When** 멤버 Y의 더 늦은 변경이 같은 행을 덮어쓰면
- **Then** X에게 데이터 손실 없이 최신 값을 안내하는 비파괴적 알림(토스트 등)이 표시된다.

### AC-NFR-RESP-01 — 모바일 반응형 레이아웃 (NFR-RESP-001)
- **Given** 360px 폭의 모바일 뷰포트에서
- **When** 지도/일정/스탬프 화면을 표시하면
- **Then** (a) 페이지에 가로 스크롤바가 나타나지 않고, (b) 지도/일정 목록/스탬프 목록이 모두 화면 폭 안에 표시되며, (c) 주요 버튼이 잘리지 않고 보인다.

### AC-NFR-RESP-02 — 모바일 터치 조작 (NFR-RESP-002)
- **Given** 360px 폭의 터치 디바이스에서
- **When** 마커 탭, 순서 이동(드래그/버튼), 일반 버튼을 조작하면
- **Then** (a) 마커 탭으로 요약이 열리고, (b) 순서 이동이 반영되며, (c) 버튼 탭이 의도한 동작을 수행한다.

### AC-NFR-SEC-01 — 비밀 키 미노출 (NFR-SEC-001, NFR-SEC-002)
- **Given** 저장소와 클라이언트 빌드 산출물에서
- **When** 소스/번들을 점검하면
- **Then** 저장소에 커밋된 비밀 값이 없고, Supabase service-role key가 클라이언트 번들에 포함되어 있지 않다. (anon key/Kakao JS key의 클라이언트 노출은 허용됨, A13)

### AC-NFR-SEC-02 — 작업별 RLS 적용 (NFR-SEC-004)
- **Given** RLS 정책이 적용된 상태에서
- **When** (a) 비멤버가 다른 투어의 spots를 SELECT 시도, (b) 멤버가 타인의 user_id로 stamps INSERT 시도, (c) 일반 멤버가 spots DELETE 시도를 하면
- **Then** 각 작업이 정책에 의해 거부되어 데이터가 반환/변경/삭제되지 않는다.

### AC-NFR-SEC-03 — Kakao 도메인 제한 (NFR-SEC-003)
- **Given** Kakao JavaScript key에 허용 도메인이 설정된 상태에서
- **When** 허용되지 않은 도메인에서 SDK 호출을 시도하면
- **Then** 카카오 측에서 호출이 거부된다.

### AC-NFR-PERF-01 — 도착 판정→스탬프 반영 지연 (NFR-PERF-001)
- **Given** dwell-time 도착 조건이 충족된 상태에서
- **When** 조건 충족 시점부터 측정하면
- **Then** 스탬프 반영 및 화면 표시까지 사용자 체감 1초 이내로 완료된다.

### AC-NFR-PERF-02 — 도심 자동 스탬프 성공률 (NFR-PERF-003)
- **Given** 도심(고층 빌딩 등 GPS 간섭) 환경에서 권한 허용 + 정확도 조건이 충족된 다수의 도착 시나리오에서
- **When** 반경 진입 후 dwell-time 동안 측정하면
- **Then** 정상 자동 스탬프 부여 성공률이 90% 이상이다. (테스트: 시뮬레이션/필드 샘플로 측정, 미달 시 dwell-time/정확도 임계값 재조정)

## Quality Gate Criteria (품질 게이트)

- spec.md의 모든 요구사항(REQ-F1~F6, NFR-*)이 Traceability 표를 통해 1개 이상의 AC에 매핑됨
- 핵심 엣지 케이스 EC-01~EC-13 검증됨
- 모든 AC가 관찰 가능한(observable) 결과만 사용(주관적 "정상 동작" 표현 제거)
- TRUST 5: 테스트 커버리지 기준 충족, 비밀 키 미노출(Secured), 일관된 스타일(Unified)

## Definition of Done (완료 정의)

- [ ] F1~F6 각 기능의 수용 기준 전부 통과
- [ ] NFR(위치 권한/추적/보존, 행 단위 충돌 해결, 반응형, 키 관리/RLS, 성능) 수용 기준 통과
- [ ] 엣지 케이스 EC-01~13 처리 확인
- [ ] 저장소/번들에 비밀 키 미포함 확인(anon/Kakao JS key 노출은 허용)
- [ ] 작업별 RLS(SELECT/INSERT/UPDATE/DELETE) 정책 검증
- [ ] 장소 삭제 cascade + 순서 재계산, 스탬프 취소/재스탬프 동작 확인
- [ ] 모바일(360px) 및 데스크톱에서 핵심 플로우 동작 확인
- [ ] 실시간 동시 편집/동시 순서 변경 시나리오(2명 이상) 검증
- [ ] 도심 자동 스탬프 성공률 90% 목표 측정
